Lessons learned from Develop Chatbot

One-shot conversation/Turn-taking conversation.

One-shot conversation, user gives an input, expect certain result and conversation ends. For example:

Q: What's the time?

A: It's 10:30 pm.

Turn-taking conversation, more specially, conversation flow. For example:

User: I want to book a flight .

Bot: Where are'u going?

User: visit Amsterdam.

This could be way more complicated than one-shot.

Lessons learned 1, keep conversation flow simple. Reduce turns otherwise user might loose patient.

Lessons learned 2, motivate/encourage user with certain words. Such as "Great!", "Nice!", "Thanks for your message!".

Lessons learned 3, user certain type of design to indicate the system is listenning/responding.

Lessons learned 4, response diversification. To make conversation more nature:

*bad

User: What's the time?

Bot: 10:30 pm.

User: What's the time?

Bot: 10:31 pm.

*Better

User: What's the time?

Bot: It's 10:30 pm now!

User: What's the time?

Bot: It's half past ten.

Lessons learned 5, response diversification without losing the focus.

*Best (as far as I know)

User: What's the time?

Bot: It's 10:30 pm now!

The idea is to use diversified response to make conversation more nature, while use certain type of design (such as bold, italic) to highlight the core part of answer.

Possible solution: 1. Response templates 2. Neural networks & Nature language generation.

Difficulties of Conversational Bot

Use wit.ai, api.ai or develop your own? Rely on such platform might make everything much easier, but rely on third party platforms might leverage certain risks.

User pattern matching or machine learning/deep learning? Pattern matching is easier to implement and seems easier to use. However, nature language and dialogue is something difficult to express with patterns (two reasons: 1. user input is so different 2. user might not explicitly express their intents in the text).

Lessons learned 6: for small companies, use third-party platform, for large companies, confidential related companies, develop your own.

Lessons learned 7: Always keep an eye on cutting-edge AI developments.

Uncertainty intent. In a real-world scenario, intent recognition always return an list of predict intents with certain probability. For example:

*INTENT ENGINE:

INTENT_FIRST: recongize as a numeric number of 1.

INTENT_SELECT_FIRST: recognize as user want to select the first option.

*CONVERSATION:

Bot: Great! Which offer would you like to choose? 1. 500 mb 5 EU per month. 2. 1 GB 8 eu per month. 3. unlimited 20 eu per month.

User: the fisrt offer.

*BACKEND

MESSAGE RECEIVED: "The first offer". Intent predicted as: [{"Intent_first": 0.33, "Intent_select_first": 0.67}]

It seems "Intent_first" received a lower probability than "intent_select_first", however, "Intent_first" also received a relative high probability. Which is dangerous!

Lessons learned 8: Carefully design intent engine, try to maximize the margin between similar intents!

Lessons learned 9: For certain type of intents can not get rid of, use explicit message/design to get confirm from user. Two important points,1. careffuly set threshold 2. Do not forget lesson one.

Intent in conversation flow could be crucial. It has to follow certain state and track context. For example:

User: Can you help me create an flight booking?

*ENGINE: INTNET_BOOK_FLIGHT (intent recognition)

*STATE: STATE_BOOK_FLIGHT_ASK_DESTINATION (expect next state)

*CONTEXT: {"INTENT": "INTENT_BOOK_FLIGHT", "STATE": "STATE_BOOK_FLIGHT_ASK_DESTINATION"} (track context)

Bot: What's your expect destination? (Successful recognize intent, trigger state and keep track context)

….

Bot: Do you prefer first class or second class?

User: 2nd class.

*ENGINE: INTENT_CLASS_SELECTED

*STATE: STATE_BOOK_FLIGHT_ASK_INSURANCE

*CONTEXT: {….."FROM": "AMSTERDAM", "TO": "BEIJING", "CLASS": 2}

BOT: Do you want to buy insurance or not? (everything seems fine now, but..)

USER 1: HAHAHAHA SOME TEST MESSAGE! HAHAHAHA

USER 2: I want to book a bus ticket.

Clearly, at the moment systems is waiting for (User case 1) YES/NO response, what if the user input of out of "expectation"?

Similarily, the system is waiting for (User case 2) YES/NO response, however, the user want to quit the current conversation flow and re-start a new one. The intent "INTENT_BOOK_BUS" can be correctly identified, however, how to deal with the current conversation flow?

Or even, what if the user just stop typing (maybe he has something else to do)?

Lessons learned 10: handle exception message just like what we have done in programming.

Lessons learned 11: Think of a mechanism to jump out current conversation flow if new intent it recognised (which is not the message (state) we are expecting) .

Lessons learned 12: Do not clear state/context unless user explicitly confirm he wanna restart. Because user might:

User: Lets continue..

Bot: Great! Do you want to buy insurance for your flight?

Lessons learned 13: remind user if he want to continue.:

User: Let's continue

Bot: Great! You're interested in a flight from Amsterdam to Beijing at 10:AM AMS time on Nov 10th, 2017 for 1 person, 2nd class. Do you want to buy insurance for your flight?

Lessons learned 14: for entity extraction, follow lessons learned 7!. For example:

Bot: What is your destination?

User: I want to go to Beijing.

This is a simple example, we want to extract BEIJING as destination, pattern matching might help, but seems dangerous. Name Entity Recognition/ Keyphrase extraction seems to be a more general idea.

Lessons learned 15: Patient, tolerate mistakes and keep optimize. Track conversation in database, analyze and keep optimise (especially the failure cases).
